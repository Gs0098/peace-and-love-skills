## 触发条件
Agent 收到一个现有 Skill，需要精简、重构、瘦身时，使用本 Spec。

---

## 核心思想

Skill 是**执行契约**，不是说明书。精简的本质不是减法，而是提纯——用最少的描述传递最多的信息。目标是：去掉所有不必要的内容，让输入/流程/输出尽可能短，但功能、逻辑、结果完全不变。

### 四条精简原则

1. 精简的本质不是减法，而是提纯——用最少的描述传递最多的信息
2. 对已经最优压缩的内容，任何压缩都是破坏
3. 删除"模型不需要知道"的内容，只保留"模型做正确决策必须知道"的内容
4. 去掉所有不必要的信息，让输入/流程/输出尽可能短，但功能、逻辑、结果完全不变

### 四层产物模型（兜底分类）

| 层级                     | 含义                   | 压缩策略                 | 示例                                     |
| ------------------------ | ---------------------- | ------------------------ | ---------------------------------------- |
| 🏗️ **Skeleton（骨架）**   | 输出容器的结构框架     | **行数/字段不压缩**      | 表格的行列、清单编号、模板字段列表       |
| 🧱 **Component（部件）**  | 容器内的具体值         | **保留核心值，压缩修饰** | 表格 cell 内的关键数字、判断条件、枚举值 |
| 🧥 **Cladding（覆盖层）** | 对结构的解释和注释     | **大幅压缩**             | cell 内的解释文字、列/行的小标题说明     |
| 🧹 **Scaffold（施工架）** | Agent 理解用的背景说明 | **可以整段删除**         | "为什么这样做"、过渡语、重复强调         |

### 判断顺序（v15 核心变更）

对每段待删内容，依次通过三层判断：

**层级 1：这段还有压缩空间吗？**
- 已最精炼（每个字独立承担语义功能，删任一字则意思模糊或缺失）→ **保留**
- 还有冗余 → 进入层级 2

**层级 2：模型少了这个信息，会做错决定吗？**
- **会 → 保留**
- 不会 → 进入层级 3

**层级 3：删了输出会变吗？**
- **会变 → 保留**
- **不变 → 删除**
- 不确定 → 进入四层分类（兜底）

**四层兜底（仅对"不确定"的内容）：**
- Skeleton → 不删
- Component → 保留核心值
- Cladding → 可大幅压缩
- Scaffold → 可删除

**全部通过 → 删除。任一不通过 → 保留。**

## 精简规则

### 规则 1：删除角色
❌ 你是一名 / 请扮演 / 请作为
Agent 身份由 Agent 层统一管理。

### 规则 2：删除冗余
同一语义只保留一次。
- 措辞不同但意思相同 → 删
- 措辞相近但意思不同 → 保留

### 规则 3：删除优先级强调标签
"最高优先级""贯穿全流程""核心原则"等修饰词，若规则本身已表达该意思，则删除。

### 规则 4：精简表达
短句、列表、结构化。去背景、去解释、去礼貌用语。每行只表一个动作或一个判断。

**结构性表格不压缩行数——保留所有行列，仅压缩单元格内的解释文字。**

### 规则 5：参数化（L1 推荐，可降级）
可参数化的判断 → 显式分支：
- ❌ 如果订单较多则批处理
- ✅ order_count > 1 → batch_flow
不可参数化的保留自然语言，但精确到唯一语义。

参数定义：
| 字段     | 说明                                     |
| -------- | ---------------------------------------- |
| Name     | 参数名                                   |
| Source   | User / Context / Tool / System / History |
| Type     | string / int / bool / enum / object      |
| Required | 是 / 否                                  |
| Rule     | 取值 → 分支映射                          |

阈值由业务人员定义，可配置。

### 规则 6：Skill 互调（L2 进阶）
允许。写在 Business Steps 中，声明调用目标 Skill、传入参数、结果回传后的分支方向（终止 / 继续）。仅同步调用。

### 规则 7：话术内容不精简
话术内容由业务人员定义，Agent 不改写。
仅检查每种话术的触发条件是否唯一、无歧义。有歧义时标注问题，但不改写话术本身。

### 规则 8：合规等级
| 等级 | 含义   | 缺省策略         |
| ---- | ------ | ---------------- |
| L0   | 必满足 | 不满足不可发布   |
| L1   | 推荐   | 不做需有等价兜底 |
| L2   | 进阶   | 按业务重要性选做 |

## 典型不得删除结构（直接保留，不过校验层）

- 反例模式（Anti-Pattern / Red Flags / Rationalizations）
- 单句生效的边界保护（Hard Gate / Iron Law）
- 结构性表格的框架行（表头、行标识、列标签）
- Output Artifacts 表的最终输出承诺
- 主干流程步骤（Phase/Step 中的动作指令，含 exit criteria）
- 业务规则与阈值（数值标准、判断条件、枚举值、路由规则）
- 正误对比示例（❌错误做法 → ✅正确做法）
- Why / Rationale 段落（流程步骤间的逻辑连接）
- Confidence Key（可信度标注：🟢 / 🟡 / 🔴）
- 检查清单的条目编号
- 决策矩阵的维度标签
- 输出模板的字段列表
- **⚓ 锚点（Anchor）：对关键规则/条件的一句话强调或后果说明。删了之后，Agent 可能对该条件的严格核对放松，导致事实性错误。**

> 校验层优先于规则 2（删除冗余）和规则 3（删除优先级强调标签）。
> 措辞与主干重复但拦截特定失误的，不算冗余，不删。

## 处理步骤

1. 逐章节阅读原 Skill
2. **锚点扫描（必做）**：逐句检查，对关键规则的强调或后果说明标记为锚点保留。这些内容删了之后，Agent 可能放松对该条件的仔细核对，导致事实性错误。**后续所有步骤跳过已标记锚点。**
3. 按精简规则 1–8 逐条对照，记录不符合项（跳过已标记锚点）。应用规则 2/3 前，先过校验层
4. 对未被标记锚点的内容，执行三层判断：
   - **层级 1：压缩空间检测**——已最精炼则保留
   - **层级 2：决策依赖判断**——模型少了这个信息会做错决定吗？会则保留
   - **层级 3：输出影响判断**——删了输出会变吗？会变则保留，不变则删除，不确定则四层兜底
5. 四层分类兜底（仅对"不确定"的内容）：
   - Skeleton → 不删
   - Component → 保留核心值
   - Cladding → 可大幅压缩
   - Scaffold → 可删除
6. 生成精简后的 Skill（按原结构输出，章节顺序不变）
7. **输出闸**：精简版返回前，检查是否仅包含规定内容。禁止输出：分析过程、失败原因、过渡语、内部逻辑。
8. 输出修改建议报告：列出每处修改的位置、原文、修改后、违反了哪条规则、判定层级。

## 自检标准

对每段被标为"待删除"的内容，依次执行：

1. **压缩空间检测**：这段还有压缩空间吗？
   - 已最精炼 → 保留
   - 还有冗余 → 进入步骤 2

2. **决策依赖判断（层级 2）**：模型少了这个信息，会做错决定吗？
   - 会 → 保留
   - 不会 → 进入步骤 3

3. **输出影响判断（层级 3）**：删了这句 Agent 的输出会变吗？
   - **会变 → 保留**
   - **不变 → 删除**
   - 不确定 → 进入步骤 4（四层兜底）

4. **四层兜底（仅对"不确定"）**：
   - Skeleton → 不删
   - Component → 保留核心值
   - Cladding → 可大幅压缩
   - Scaffold → 可删除

**全部通过 → 删除。任一不通过 → 保留。**

> 本条优先级高于规则 2 和规则 3。
> 锚点内容跳过校验，直接保留。

## 输出闸
精简版返回前，检查是否仅包含规定内容。禁止输出：分析过程、失败原因、过渡语、内部逻辑。

## 边界
本 Skill 不负责：原 Skill 的业务正确性判断、最终用户回复话术撰写、调用其他 Skill 做补充分析。
